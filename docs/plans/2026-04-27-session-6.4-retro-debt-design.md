# Session 6.4 — Retro debt cleanup before Session 7

**Date:** 2026-04-27
**Predecessor:** Session 6.3 (Flux workflow rebuild + subject gate; closed 2026-04-26 with Cask scenes 8 + 16 viable, scene 1 deferred).
**Successor:** Session 7 (full Cask 16-scene keyframe run + Wan 2.2 I2V video stage).

## 1. Context and motivation

Session 6.3 Phase 2 shipped on a single A6000 rental (~$1.30) and closed with two of three eye-check scenes (Cask 8 + 16) viable. Closure threshold (≥2 of 3) was met, but the Phase 2 retro surfaced four pieces of debt that must clear before Session 7 chains into the full Cask 16-scene run plus the new Wan 2.2 I2V video stage:

1. **Cask scene 1 visual_prompt is too dark-loaded for the workflow** — mean_rgb stays at 2.6–3.0 even on the rebuilt FluxGuidance + ModelSamplingFlux workflow with `steps=60`. Brightness floor was lowered to 5.0 in S6.3, but scene 1 is still under it. The visual_prompt itself stacks darkness modifiers ("dark velvet doublet, candlelit study at night, single oil lamp, the other half lost in shadow, dissolving into blackness, low-key chiaroscuro, muted desaturated palette") and the negative_prompt actively forbids the available light source ("multiple light sources"). Flux renders the prompt faithfully — the prompt itself describes a near-imperceptible image. Without a fix, S7's full Cask run halts at scene 1 and never reaches the Wan video stage.

2. **`track.image_model.candidates_per_scene` is a no-op.** Yaml says 3, runtime defaults to 3, no observable bug — yet. But S7's first task is "is 5 candidates per scene enough to drive selection consistency given the new gates?" That experiment requires the yaml override to actually take effect.

3. **`reset_scene_keyframes.py` doesn't strip `keyframe_generator` StageRun records.** Per the S6.3 Phase 2 runbook entry: setting `keyframe_path = None` per-scene is necessary but not sufficient — the orchestrator's `Stage.is_complete()` reads `story.latest_stage_run("keyframe_generator")` from `story.json["stages"]`, and a prior COMPLETE record short-circuits the entire stage in <5s without entering `generate()`. Currently mitigated with a hand-rolled Python one-liner in the runbook. S7 needs >1 keyframe re-run per rental (calibration loops); a clean reset primitive saves time.

4. **`vast_setup.sh` doesn't provision Python 3.11.** vast.ai's `pytorch/pytorch:latest` image ships Python 3.10. Platinum's `pyproject.toml` requires `>=3.11` (uses `enum.StrEnum`, `datetime.UTC`, plus transitive numpy 2.x / contourpy 1.4+ / matplotlib 3.10+ that all demand 3.11). S6.3 Phase 2 burned ~30 min on the polyfill dance (sitecustomize.py with StrEnum and `datetime.UTC` shims, `pip install --ignore-requires-python`, version-pinned dep list). The runbook entry already names the proper fix: `conda create -n p311 python=3.11 -y`.

The four items are independent. Items 2, 3, 4 are pure offline TDD or script edits (no user gates required — orchestrator-driven by spec). Item 1 has two explicit user gates (template review pre-Claude-call; eye-read post-Claude-call). All items together fit in a single self-driven session under a $5 budget cap.

## 2. Brainstorming forks (decisions logged)

Captured here for the same reason every other Platinum session captures them: future-me re-reading this in S7+ needs to know not just *what* was decided but *why*, so edge cases can be judged rather than re-litigated.

### Fork 1: Item 1 scope — re-run all of `adapt`, only `visual_prompts`, or hand-author scene 1?

| Option | Cost | Risk |
|---|---|---|
| (a) Full `platinum adapt` (story_adapter + scene_breakdown + visual_prompts) | ~$1.18 | Regenerates the user-approved Session 4 narration_script + scene structure — a regression risk, since those are already locked-in artifacts |
| (b) **visual_prompts stage only** | ~$0.40 | Regenerates all 16 scenes' visual_prompts, including the already-viable 8 + 16 — mitigated by `cp story.json story.json.s64-backup` before the re-run, with diff inspection at Gate 2 |
| (c) Hand-author scene 1's visual_prompt directly | ~$0 | Loses the "test the new template" benefit; one-off fix that doesn't validate the cap rule works generally |

**Chose (b).** The visual_prompts stage is implemented as a single Claude call that takes all scene narration_text values and emits per-scene visual_prompt + negative_prompt. The orchestrator's `Stage.is_complete()` is per-stage; stripping only the `visual_prompts` StageRun lets the next `platinum adapt` re-enter only that stage (story_adapter + scene_breakdown remain COMPLETE and are skipped). The diff inspection at Gate 2 catches scene 8/16 regressions; the cap rule is *additive*, so catacomb scenes (which already comply because they describe torch-lit interiors with subjects up front) should diff minimally if at all.

### Fork 2: Item 3 default — opt-in flag or always-strip-on-reset?

**Chose opt-in (`--reset-stage` boolean, default OFF).** There are legitimate cases — re-run one scene of a partially-completed batch — where the user wants to clear `keyframe_path` for a single scene without invalidating the entire stage's StageRun history. Default OFF preserves that. Default ON would silently break those cases, and the per-rental smoke loop is the only known caller anyway, where opt-in is one extra flag for a clear gain in invariant clarity.

### Fork 3: Item 3 strip semantics — strip all `keyframe_generator` runs or only the latest?

**Chose strip all.** Simpler invariant ("zero keyframe_generator runs after `--reset-stage`"), and history of failed runs isn't useful once we're re-running clean. Matches the runbook's manual fix recipe verbatim. Only-latest would require comparing creation timestamps and would still leave older COMPLETE records that confuse `latest_stage_run` if the order is wrong.

### Fork 4: Item 4 install depth — script also installs platinum into `p311`, or just creates the empty env?

**Chose install.** The runbook flow becomes `git clone → bash vast_setup.sh → done` instead of `git clone → bash vast_setup.sh → conda activate p311 → pip install -e . → done`. The `pip install -e /workspace/platinum` block runs *late* in the script (after ComfyUI / score-server bring-up), so a platinum install failure doesn't block earlier work. The script is idempotent — `conda env list | grep p311` short-circuits creation, and `pip install -e` updates the existing install in place.

### Fork 5: Sequencing — phases

**Phase A (no user gates, self-driven):** Items 2 → 3 → 4 in priority order (matches user spec). Each is its own commit on `main`. Tests pass, ruff clean, mypy unchanged from S6.3 baseline.

**Phase B (with user gates):** Item 1 last, gated as below. Phase A could parallelize with Phase B's Gate-1 wait, but the gain is small (Phase A is ~1-2 hours of self-driven TDD; Gate 1 is asynchronous). Keep them sequential to avoid context fragmentation.

### Fork 6: Strip visual_prompts StageRun via inline Python or generalized `--reset-stage <stage_name>`?

**Chose inline Python.** Item 3's `--reset-stage` flag is hardcoded to `keyframe_generator` per user spec; generalizing to take an arbitrary stage name conflicts with the script's identity ("reset keyframe state for specified scenes"). Item 1's visual_prompts re-run uses an inline 6-line Python heredoc instead. The orchestrator-skip workaround for visual_prompts is documented in this design and the eventual S6.4 closeout commit message; future stage-specific resets (S8 video, S13 audio) can either copy the inline pattern or get their own targeted reset script.


```python
         "Stage.is_complete().",

    *, story_id, scenes, story_dir, delete_files, reset_stage=False,

    if reset_stage:

            if s.get("stage") != "keyframe_generator"
        ]
        stripped = before - len(data["stages"])
        print(f"stripped {stripped} keyframe_generator StageRun(s)", flush=True)
    # existing atomic write...

# main() forward:
return reset_scenes(
    story_id=args.story_id,
    scenes=scenes,
    story_dir=args.story_dir,
    delete_files=args.delete_files,
    reset_stage=args.reset_stage,
)
```
