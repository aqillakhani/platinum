## Why this exists

S8.A closed at 15/16 + 1 borderline on Wan I2V *motion* coherence (commit `fd146ab`).
But the user's eye-check at the end of S8.A flagged a deeper problem the
motion fix could not address: **the keyframes themselves don't depict the
specific narrative beat of the story.**

Sample complaints (S8.A end review, voice-transcribed):
- Scene 1: "no candle flicker, doesn't feel like he's brooding"
- Scene 2: "looks like grandpa carrying a son, no eye contact between them"
- Scene 3: "no bells, hat is wrong"
- Scene 7: "floating bottle"
- Scene 9: "wrong character — that's not the moment from the story"

The pattern: render quality is fine. Motion coherence is fine (post-S8.A).
**Content fidelity to the story is broken.** Flux drifts to a generic "moody
horror" prior because the per-scene `visual_prompt` is generated from
`narration_text` alone — the prompt writer never sees the rest of the story
or the character continuity it needs to anchor to.

S7.1 patched the prompt structure (camera framing, active verbs, 9:16
portrait). It did not fix the architectural gap: every scene is generated in
isolation, so the writer cannot enforce continuity, character identity,
spatial layout, or the iconic Poe blocking that makes a beat *recognisably
the right beat*.

The S8.B prototype (commit-less; artifacts in `data/s8B_*`) validated the
fix on real Flux. A two-step pipeline — Opus 4.7 reads the whole story and
produces a structured **story bible**, then Sonnet 4.6 rewrites each
visual_prompt grounded in that bible — moved scene 9 from a confused
single-figure render to the iconic two-character catacomb staging, and
scene 3 from a generic carnival meeting to one with the conical jester
cap. Two of four scenes were clear wins; one was a content win with an
exposure regression (Flux too dark); one was a regression (rewriter dropped
a character). The prototype identified two fixable failure modes — both
addressed in this design.

By end of session:

- A new `story_bible` pipeline stage produces a structured JSON bible from
  the full story (one Opus call, ~$0.30/story).
- The bible is persisted in `Story.bible` and round-tripped via `story.json`.
- The `visual_prompts` stage is rewritten to consume the bible; per-scene
  prompts lead with `hero_shot` + `visible_characters` + `props_visible`.
- The visual_prompts template gets two **guardrails** to prevent the
  prototype's regressions:
  1. **Exposure floor.** Rewriter must include at least one named light
     source per scene; banned-list cannot exclude all light. A new
     `brightness_floor` hint per scene constrains how dark the rewriter
     can push.
  2. **Character enumeration.** The bible declares `visible_characters` per
     scene; rewriter post-condition validates every name in
     `visible_characters` appears in the rewritten prompt by lexical match;
     mismatch raises `ClaudeProtocolError` and triggers one retry.
- A bug found during the prototype is fixed: `KeyframeGeneratorStage`
  silently skips re-runs once `stage_run.status == COMPLETE`, even when
  `--scenes` filter targets scenes whose `keyframe_path` is `None`.
  Stage's `is_complete` is overridden to be per-scene-aware.
- One A6000 verify run on the Cask story closes the session if eye-check
  ≥14/16 scenes are content-faithful with no exposure or character-drop
  regressions.

## Architecture

### Pipeline placement

```
story_curator → story_adapter → scene_breakdown → STORY_BIBLE → visual_prompts
                                                  (NEW)         (rewritten)
              → character_references → pose_depth_maps → keyframe_generator → ...
```

`StoryBibleStage` is inserted between `scene_breakdown` and `visual_prompts`.
It is **opt-in per track** (default off; atmospheric_horror = on; others off
until validated). When off, `visual_prompts` runs unchanged on
`narration_text`-only inputs.

### Data model

A new `StoryBible` dataclass attached to `Story.bible: StoryBible | None`.
Round-tripped via existing `Story.to_dict` / `Story.from_dict`. Existing
story.json files load with `bible=None` (default).

```python
@dataclass(frozen=True)
class BibleScene:
    index: int
    narrative_beat: str                 # "Montresor first sees Fortunato in the carnival crowd"
    hero_shot: str                      # "two-shot, medium, eye-level, 8m apart closing"
    visible_characters: list[str]       # ["Montresor", "Fortunato"]
    gaze_map: dict[str, str]            # {"Montresor": "tracking Fortunato", "Fortunato": "into camera, drunk"}
    props_visible: list[str]            # ["jester bells", "torch flame", "Montresor's hidden mask"]
    blocking: str                       # "Fortunato foreground-right, swaying; Montresor mid-ground-left, still"
    light_source: str                   # "lantern overhead carnival, warm amber"
    color_anchors: list[str]            # ["motley red+yellow stripes", "Montresor's black cloak"]
    brightness_floor: str               # "low" | "medium" | "high" (drives exposure guardrail)


@dataclass(frozen=True)
class StoryBible:
    world_genre_atmosphere: str         # short paragraph
    character_continuity: dict[str, dict[str, str]]
                                        # {"Montresor": {"face": ..., "costume": ..., "posture": ...}, ...}
    environment_continuity: dict[str, str]
                                        # {"palazzo_study": "ash-grey wall, oak desk", ...}
    scenes: list[BibleScene]            # one entry per Story.scene, indices match
```

Persistence: the bible is one new top-level key in story.json (`"bible"`),
adjacent to existing keys (`scenes`, `stages`, etc.). No separate file.
This keeps `is_complete` resume semantics simple (one file = one
checkpoint).

### Stage interaction

1. `StoryBibleStage.run()`: one Opus 4.7 call with the full
   `Story.scenes[].narration_text` + adapted source as input. Returns
   structured JSON via tool-use (`submit_story_bible` tool). Persisted to
   `Story.bible` and saved.

2. `VisualPromptsStage.run()`: existing single Sonnet 4.6 call, but the
   rendered prompt template (`visual_prompts.j2`) now reads
   `Story.bible` if present and per-scene injects:
   - `bible.scenes[i].hero_shot`
   - `bible.scenes[i].visible_characters` (names only)
   - `bible.scenes[i].props_visible`
   - `bible.scenes[i].brightness_floor`
   - `bible.character_continuity[char]` for each visible character
   - `bible.environment_continuity[env]` for each scene's locale
   The template instructs Sonnet to lead with `hero_shot`, name every
   character in `visible_characters` explicitly, include every prop in
   `props_visible`, and keep the brightness floor as a constraint when
   choosing light language.

3. `_zip_into_scenes()` post-condition (new): for each rewritten
   scene, every name in `bible.scenes[i].visible_characters` must appear
   as a substring in the new `visual_prompt` (case-insensitive). On
   mismatch → raise `ClaudeProtocolError`. The orchestrator's existing
   one-retry path covers this.

## Changes

### 1. New stage: `src/platinum/pipeline/story_bible.py` (~280 lines)

Mirrors `visual_prompts.py` exactly. Files & sections:

- `STORY_BIBLE_TOOL`: tool-use schema (one tool, one call, one struct
  matching `StoryBible` shape above).
- `_build_request(*, story, track_cfg, prompts_dir)`: renders
  `system_bible.j2` + `story_bible.j2`. User message contains all scene
  narrations + adapted source.
- `_zip_into_story(story, tool_input)`: validates schema; constructs
  `StoryBible` dataclass; sets `story.bible`.
- `async story_bible(*, story, track_cfg, prompts_dir, db_path,
  recorder=None)`: top-level coroutine. Reuses
  `platinum.utils.claude_call` (the same helper visual_prompts uses).
- `class StoryBibleStage(Stage)`: name=`story_bible`. `run` calls
  `story_bible()`, persists, returns artifacts.
  `is_complete` override: `story.bible is not None and len(story.bible.scenes) == len(story.scenes)`.

Cost per story: Opus 4.7 input ~12K tokens (Cask = 16 scenes × ~700 chars
= ~11K + adapted source) → ~$0.18. Output ~6K tokens → ~$0.30. Total ~$0.30.
The prototype validated 16K `max_tokens` is enough for the full bible.

### 2. Story / Scene model updates

`src/platinum/models/story.py`:
- Add `BibleScene` and `StoryBible` dataclasses (frozen).
- Add `Story.bible: StoryBible | None = None`.
- Update `Story.to_dict` / `Story.from_dict` to round-trip the bible.
- Existing story.json without `"bible"` key loads with `bible=None`.

`src/platinum/models/story_bible.py` (new file): the dataclasses live
here; `story.py` imports for the Story field. Keeps `story.py` from
ballooning.

### 3. Visual prompts rewriter

`config/prompts/atmospheric_horror/visual_prompts.j2`:
- Add a top-level `STORY BIBLE CONTEXT` block (only rendered when
  `bible` is in template context).
- Per-scene block now includes: `hero_shot`, `visible_characters`,
  `props_visible`, `brightness_floor` directives.
- The leading instruction becomes: "Lead the prompt with the hero shot.
  Name every character in `visible_characters` explicitly. Include every
  prop in `props_visible`. The output must be 60–140 words. **Brightness
  floor: see per-scene; never produce an output that bans every light
  source.**"
- Negative-prompt guardrail: a track-level minimum negative does not
  exclude `candle`, `torch`, `lantern`, `flame`, `light source` (the
  prototype's scene 1 banned every light element and Flux drove RGB to ~2.0).

`src/platinum/pipeline/visual_prompts.py`:
- `_build_request` reads `story.bible` and threads bible context into
  the template.
- `_zip_into_scenes` runs the new visible-characters post-condition.
- `VisualPromptsStage.is_complete` adds: if `track_cfg.story_bible.enabled`,
  require `story.bible is not None` (otherwise the stage cannot meet
  its post-condition; the orchestrator should fail fast rather than
  silently skip).

### 4. Track configuration

`config/tracks/atmospheric_horror.yaml`:
```yaml
story_bible:
  enabled: true
  model: claude-opus-4-7
  max_tokens: 16000
visual_prompts:
  model: claude-sonnet-4-6   # explicit; was implicit before
```

Other four tracks add the block with `enabled: false`. They run unchanged
until per-track validation.

### 5. CLI plumbing

`src/platinum/cli.py`:
- New `platinum bible <story_id>` command. Runs only `StoryBibleStage`.
  `--rerun` flag clears `story.bible` and regenerates.
- `platinum keyframes` (existing): orchestrator gains `StoryBibleStage`
  ahead of `VisualPromptsStage`. If track's `story_bible.enabled = false`,
  the stage is omitted from the orchestrator list (track-aware
  composition).
- `platinum visual-prompts <story_id>` (existing): if track has
  `story_bible.enabled = true` AND `story.bible` is None, fail with a
  clear error pointing at `platinum bible`.

### 6. KeyframeGeneratorStage `is_complete` fix

`src/platinum/pipeline/keyframe_generator.py`: override the base
`is_complete` to be per-scene-aware. Returns False if any scene with
`keyframe_path is None` is in the active scene_filter (or in all scenes
when no filter).

```python
def is_complete(self, story: Story) -> bool:
    runtime = ...  # config.settings.get("runtime", {})
    scene_filter = runtime.get("scene_filter")
    targets = (
        [s for s in story.scenes if s.index in scene_filter]
        if scene_filter else story.scenes
    )
    return all(s.keyframe_path is not None for s in targets)
```

This was a real bug (cost two GPU re-launches during the S8.B prototype).
Worth fixing in the same session even though it's not the headline
feature.

2. Reset `story.bible` to None on the Cask story (one-line script) so
   we exercise the new path end-to-end.

   `platinum keyframes <id> --rerun-all`).
4. `platinum bible story_2026_04_25_001` (~$0.30, ~30 s).
5. `platinum keyframes story_2026_04_25_001` (~$0.55 Claude + ~$0.50
   Flux, ~7 min).
6. `platinum review keyframes story_2026_04_25_001` — eye-check, mark
   approve / reject / regenerate per scene.
7. Re-render rejected scenes via `--rerun-rejected`; iterate until
   ≥14/16 approved.

Total verify budget: ~$1.50 cloud + 1 hr GPU = ~$1.90, well under $5
abort.

## Closure criteria

S8.B is closed when:
- ≥14/16 Cask scenes pass user eye-check on **content fidelity**
  (correct characters, correct props, correct beat) — same bar S7.1 set.
- All 16 keyframes have `mean_rgb ≥ 5.0` (current track floor) — i.e.
  no exposure regressions vs. pre-bible baseline.
- All 16 keyframes name every `visible_characters` entry in their
  `visual_prompt` (post-condition asserted).
- Tests green; mypy clean; ruff clean.
- Commits pushed to `origin/main`.

If <14/16 after one round of `--rerun-rejected`, escalate to S8.C
(per-scene bible refinement loop, or character-reference re-render via
S7.2 deferred Phase B). Don't widen scope this session.
