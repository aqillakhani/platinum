## Why this exists

Phase 2 verify shipped 16/16 Cask MP4s on A6000 ($3.75, 7hr). Eye-check
revealed two distinct quality problems in the output:

1. **Aspect stretch.** Clips are 960x544 landscape; keyframes are 768x1344
   portrait. `WanVideoImageToVideoEncode` resized the portrait keyframe
   into a landscape latent (vertical 0.40x squash, horizontal 1.25x
   stretch). The squash is baked into pixels ‚Äî not fixable in post.

2. **Narrative incoherence ("AI slop").** Wan animates the action a second
   time from a keyframe that already depicts that action mid- or
   post-completion. Examples from the Cask run:
   - Scene 5: keyframe shows masked man; prompt says "tying a black silk
     mask"; output animates putting the mask on AGAIN over an already-masked face.
   - Scene 6: prompt says "descending stone staircase carrying flambeaux";
     output shows a man walking downstairs BACKWARDS, three torches for two people.
   - Scene 7: prompt describes "handing a freshly broken wine bottle"; output
     is incoherent because three simultaneous actions don't fit in 5 s.

**Root cause** of #2: the same `visual_prompt` is reused for Flux (still
image) and Wan 2.2 (motion). Flux-style prompts describe *state at a moment*
("medium shot, X tying Y while Z..."). Wan reads them and tries to
re-perform the action from a keyframe that has already completed it.

Research review (Motion-I2V SIGGRAPH 2024, VLIPP 2025, Kijai's WanVideoWrapper
docs, ComfyUI Wan 2.2 tutorial) lines up: the field's converging on
*keyframe-grounded motion prompts* ‚Äî VLM looks at the chosen keyframe,
captions what's depicted, then writes a 5 s motion that continues from that
state without re-performing.

This session implements four changes that bundle together for the next
re-render: aspect fix, sampler tuning, negative-prompt expansion, and a new
`motion_prompts` pipeline stage.

## Changes

### 1. Aspect ratio (config-only, blast radius: track YAMLs + 1 test)

`config/tracks/atmospheric_horror.yaml`:
- `video_model.width: 960 ‚Üí 544`
- `video_model.height: 544 ‚Üí 960`
- Same pixel budget (522,240 px); both divisible by 16 (Wan VAE 8x +
  DiT patch 2x); aspect 0.567 ‚âà 9:16=0.5625.

Regression test: for every track YAML, if both `image_model` and
`video_model` declare `width`+`height`, assert orientations match (both
portrait or both landscape) and aspects within 10%.

### 2. Sampler tuning (workflow JSON only)

`config/workflows/wan22_i2v.json`:
- Both samplers (nodes 8, 9): `cfg: 6.0 ‚Üí 2.5`, `steps: 20 ‚Üí 10`
- Node 8: `end_step: 10 ‚Üí 5`. Node 9: `start_step: 10 ‚Üí 5`.

Research findings: I2V over-constrains above CFG 3 (community Lightx2v /
Phr00t guidance); step return drops sharply after 12. As a side effect,
sampling time roughly halves (20‚Üí10 steps) ‚Äî re-render budget drops from
~$2.50 to ~$1.30.

### 3. Negative prompt expansion (workflow JSON only)

Append narrative-coherence terms to node 6's `negative_prompt`:
- `duplicated action, repeated motion, action reset, looping gesture`
- `reversed motion, backward direction`
- `duplicate limbs, extra objects, floating copies`
- `morphing, warping, face deformation, flickering`

Existing render-quality negatives stay.

### 4. New `motion_prompts` stage

The architectural fix. Inserts between `keyframe_review` and
`video_generator`. For each scene with an approved keyframe:

1. Load the chosen keyframe image (base64 PNG).
2. Send to Claude Haiku 4.5 (vision-capable, ~$0.005/call) with:
   - System: motion-prompt-writing role + 5 s budget rule.
   - User: track aesthetic, original `visual_prompt` (for action context),
     scene narration text, and the keyframe image.
3. Claude returns `motion_prompt` (‚â§ 80 words) via tool-use.
4. Stage writes `motion_prompt` to `Scene.motion_prompt` and persists.

`video_generator` is updated to plumb `scene.motion_prompt or
scene.visual_prompt` into the workflow's positive prompt slot. If
`motion_prompts` was never run, the previous behaviour is preserved
(fallback to `visual_prompt`).

#### Stage architecture

Mirrors `visual_prompts.py` (a known-good template in this repo):

```
motion_prompts.py
  ‚îú‚îÄ‚îÄ MOTION_PROMPT_TOOL: tool-use schema (motion_prompt: str)
  ‚îú‚îÄ‚îÄ _build_request(...) -> system, messages with image content block
  ‚îú‚îÄ‚îÄ _zip_into_scenes(...) -> mutate scenes in place
  ‚îú‚îÄ‚îÄ async motion_prompts(...) -> tuple[list[Scene], ClaudeResult]
  ‚îî‚îÄ‚îÄ class MotionPromptsStage(Stage)
```

Vision plumbing mirrors `utils/content_check.py:ClaudeContentChecker`,
which already handles base64 image blocks via the same `claude_call`
helper.

#### Scene model change

Add to `Scene`:
```python
motion_prompt: str | None = None
```

`to_dict` and `from_dict` round-trip it. Existing `story.json` files
without the field load fine (default `None`).

#### Prompt template

`config/prompts/atmospheric_horror/motion_prompt.j2` and per-track
copies. Single template (no per-scene loop) ‚Äî one image ‚Üí one prompt
per call, per scene. Per-scene calls keep cost predictable and let
fixture replay key on `(story_id, scene_index)`.

Template content brief:
- Tell Claude: "Look at the keyframe. Describe a 5-second continuation."
- Don't re-state composition (the image speaks).
- Use motion verbs that fit 5 seconds; prefer camera language ("slow
  dolly forward") to subject state changes.
- Forbidden: actions that the keyframe has already completed, or that
  imply a state change inconsistent with the visible state.

#### CLI

`platinum motion-prompts <story_id>` ‚Äî runs `MotionPromptsStage` over
all approved scenes. Skip-if-already-set semantics for resume.

#### Cost

Per Cask story: 16 scenes √ó ~$0.005 = ~$0.08 per render. Recorder
fixtures replay free in tests.
