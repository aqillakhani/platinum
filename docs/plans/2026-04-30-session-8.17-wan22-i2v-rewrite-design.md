## Why this exists

S8 Phase A shipped `config/workflows/wan22_i2v.json` and the supporting Python
plumbing under the assumption that "live verify naturally iterates on
`class_type` drift." Phase 2 rented an A6000, ran preflight 6/6 GREEN, queued
the workflow, and ComfyUI errored at sampler step 0:

```
Given groups=1, weight of size [5120, 36, 1, 2, 2],
expected input[1, 68, 9, 30, 52] to have 36 channels, but got 68 channels instead
```

The previous post-mortem (`memory/project_s8_phase2_blocked.md`) attributed
the failure to "WanVideoImageToVideoEncode produces 2.1-shape image_embeds
that the Wan 2.2 HIGH expert can't consume" and concluded that "WanVideoWrapper
has no plain Wan 2.2 I2V example in `example_workflows/`."

**That conclusion was wrong.** Offline cross-check of the saved
`docs/research/s8-wan22-workflow/wanvideo_examples.tar` against
`comfy_object_info.json` (both pulled from the destroyed box) reveals:

1. `wanvideo_2_2_I2V_A14B_example_WIP.json` IS the canonical plain 2.2 I2V
   reference. The "WIP" suffix made it look like noise; it is not.
2. **All four** Wan 2.2 14B reference workflows (vanilla I2V, Pusa, TimeToMove,
   plus the 2.1 14B I2V baseline) load `Wan2_1_VAE_bf16.safetensors`. The Wan
   2.2 14B experts inherit the 2.1 VAE encoder; the 2.2 VAE is for the 2.2
   5B family only.
3. Plain Wan 2.2 14B I2V is a **two-sampler MoE chain** (HIGH expert runs steps
   0→split, LOW expert runs split→end with `samples` handoff), not a single
   sampler with `model_high_noise` + `model_low_noise` inputs (those inputs
   don't exist on `WanVideoSampler`).

The 36ch vs 68ch error is therefore the compound result of two errors in the
shipped workflow:

- **Wrong VAE:** loading the 2.2 5B VAE produced latent shapes the HIGH expert
  was not trained against.
- **Fictional MoE-in-one-node sampler:** the workflow wired both experts to a
  single `WanVideoSampler` via inputs that don't exist; even after class-name
  triage, the resulting single-expert graph still failed because of (a).

This document specifies the rewrite that addresses both, plus the supporting
changes to `inject_video`, `vast_setup.sh`, and `scripts/preflight_check.py`.

## Architecture — 11-node graph

Two-expert MoE chain mirroring `wanvideo_2_2_I2V_A14B_example_WIP.json`,
stripped of optional accelerators (Lightning LoRA, BlockSwap, KJNodes
helpers, ComfyUI text-encoding bypass) that we do not need on a 48GB A6000.

```
[1] WanVideoModelLoader (HIGH)  ──┐
[2] WanVideoModelLoader (LOW)   ──│──┐
[3] WanVideoVAELoader (Wan 2.1) ──┼──┼──┐
[4] LoadWanVideoT5TextEncoder   ──│──│──│──┐
[5] LoadImage  ──────────────────┐│  │  │  │
                                 ↓↓  ↓  ↓  ↓
[6] WanVideoTextEncode (t5←[4], pos+neg)              → text_embeds
[7] WanVideoImageToVideoEncode (vae←[3], image←[5],   → image_embeds
        widgets: w=1280, h=720, num_frames=81,
                 noise_aug=0, start_lat_str=1, end_lat_str=1,
                 force_offload=true, fun_or_fl2v=false)
[8] WanVideoSampler HIGH (model←[1], image_embeds←[7], text_embeds←[6],
        steps=30, cfg=6.0, shift=8, seed=⟨inj⟩, scheduler="unipc",
        end_step=15)                                  → samples_mid
[9] WanVideoSampler LOW  (model←[2], image_embeds←[7], text_embeds←[6],
        samples←[8], steps=30, cfg=6.0, shift=8, seed=⟨same⟩,
        scheduler="unipc", start_step=15)             → samples_final
[10] WanVideoDecode (vae←[3], samples←[9])            → IMAGE
[11] VHS_VideoCombine (images←[10], frame_rate=16,
        format="video/h264-mp4")                      → wan_output_*.mp4
```

### Why these specific values

- `num_frames=81` at `frame_rate=16` → 5.0625s clip, matches the runbook's
  `duration_target_seconds=5.0`.
- `cfg=6.0`, `shift=8.0`, `scheduler="unipc"`, 30 steps split 50/50 — these
  are the WanVideoWrapper defaults for vanilla (non-Lightning) 14B I2V. The
  WIP example's `cfg=1, shift=8, dpm++_sde, 6 steps` are Lightning-LoRA
  overrides; those values do not apply when the LoRA is absent.
- `width/height` written as widget values on `WanVideoImageToVideoEncode`
  (per the schema in `comfy_object_info.json`, both are required widgets).
  Keyframes are already 1280×720 post-S6.4, so we skip the `ImageResizeKJv2`
  preprocessor.
- **Seed on both samplers, identical value** — the MoE pair must use a
  coupled seed or determinism breaks at the HIGH→LOW handoff.
- `attention_mode="sdpa"` (PyTorch native). The reference uses `sageattn`,
  which requires the `sageattention` pip package; not installed and not
  worth the dependency for the closure target.

### `_meta.role` mapping

```json
"role": {
  "image_in":    "5",
  "prompt":      "6",
  "seed":        ["8", "9"],
  "video_out":   "11",
  "width":       "7",
  "height":      "7",
  "frame_count": "7",
  "fps":         "11"
}
```

`seed` is the only role that maps to multiple node IDs. All other roles
remain single strings; existing callers (Flux keyframe workflows, etc.) are
unaffected.

## Code-side changes

### `src/platinum/utils/workflow.py`

Three surgical edits to `inject_video`:

1. **New helper** — `_resolve_role_ids(workflow, role) -> list[str]`. Wraps
   single-string roles into a one-element list; passes lists through
   unchanged. Raises `KeyError` on missing role (mirroring `_resolve_role`).
2. **Seed write** — replace
   ```python
   out[seed_id]["inputs"]["seed"] = seed
   ```
   with
   ```python
   for sid in _resolve_role_ids(out, "seed"):
       out[sid]["inputs"]["seed"] = seed
   ```
3. **Frame-count field name** — replace
   ```python
   out[roles["frame_count"]]["inputs"]["length"] = frame_count
   ```
   with
   ```python
   out[roles["frame_count"]]["inputs"]["num_frames"] = frame_count
   ```

`_resolve_role` (single-id) stays — only `seed` needs list semantics.

### `config/workflows/wan22_i2v.json`

Full rewrite per the topology above. 11 nodes; `_meta.role` per the mapping
above; `_meta.comment` updated to reference S8.17 + the 2.1 VAE rationale.
Negative prompt is a tight English cinematic baseline:

> blurry, low quality, jpeg artifacts, distorted, deformed, mutated, extra
> limbs, missing limbs, watermark, text, signature, oversaturated, amateur,
> cartoon, sketch, painting, 3d render, motionless picture, still image,
> frozen frame

(Track-specific negative overrides can be plumbed through `inject_video`
later if Phase 2 surfaces a need.)

### `scripts/vast_setup.sh`

- **Remove** the `Wan2_2_VAE_bf16.safetensors` download (no longer
  referenced by any workflow).
- **Add** the `Wan2_1_VAE_bf16.safetensors` download from `$KIJAI_BASE`.
- **Add** a `ComfyUI-VideoHelperSuite` clone block (clone idempotent;
  pip-install requirements inside the ComfyUI venv; pip cache purge).

Disk delta: ≈ 0 (2.1 VAE and 2.2 VAE are both ~1.4GB; we are swapping, not
adding).

### `scripts/preflight_check.py`

- Update the `WAN_WEIGHT_FILES` manifest: `Wan2_2_VAE_bf16.safetensors` →
  `Wan2_1_VAE_bf16.safetensors`.
- Add `_check_videohelpersuite_extension_dir` — folder-existence check on
  `<comfy_root>/custom_nodes/ComfyUI-VideoHelperSuite/`. (No Python import
  barrier; node registration is the only contract.)
- Wire it into the Wan-mode check list in `main()`. Wan-mode goes 6 → 7
  GREEN; flux-mode is unchanged (still 4).

### Abort criteria

- **Probe graph error** (class-not-found / channel-mismatch / link-type /
  VAE-not-found): destroy box; re-investigate offline using
  `comfy_object_info.json`. Do not keep the box alive at $0.42/hr while
  debugging.
- **Probe MP4 generates but is all-black or all-static** for >50% of frames:
  kill probe, audit `cfg / shift / scheduler / num_frames` defaults against
  the WIP example's actual run-time settings, then retry probe. Do not
  escalate to full.
- **Cumulative spend across both rentals exceeds $5** without ≥14/16
  closure: stop, regroup, document blockers in a new memory file.

## Open risks / unknowns

1. **T5 file format.** `vast_setup.sh` currently downloads
   `models_t5_umt5-xxl-enc-bf16.pth` (Wan-AI repo, .pth) saved as
   `umt5_xxl.pth`. The WIP example references
   `umt5-xxl-enc-bf16.safetensors`. The previous Phase 2 attempt's
   sampler-step-0 error proves the .pth loaded fine (T5 encode happened
   before the sampler), so we keep the existing `.pth`. Mitigation: if probe
   surfaces a load error here, swap to Kijai's .safetensors version.
2. **`Wan2_1_VAE_bf16.safetensors` URL.** Inferred from the WIP example's
   `wanvideo\Wan2_1_VAE_bf16.safetensors` reference and Kijai's
   `WanVideo_comfy` repo layout. Verify the exact resolve URL during probe
   provisioning; if 404 then check the `Vae` subfolder pattern Kijai uses
   for newer uploads.
3. **`attention_mode="sdpa"` performance.** Reference uses `sageattn` which
   is ~2x faster but adds a pip dependency. Acceptable trade-off; if probe
   sampling is unexpectedly slow (>10 min/scene), reconsider.
4. **`riflex_freq_index=0`.** Required input on `WanVideoSampler`; default
   0 is fine for vanilla I2V (RIFLEx is for length-extension). Documented
   here so a future reader doesn't wonder why we set it explicitly.
