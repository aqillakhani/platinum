**Date:** 2026-04-28

**Spec:** S7 Phase 2 retro (the live A6000 16-scene Cask render done 2026-04-28).
**Predecessor:** S7 Phase 1 (review UI shipped 2026-04-28, last commit `aefcc5a`).

## 1. Goal & scope

S7 Phase 2 ran a full 16-scene Cask render on a fresh A6000 box. All 16 scenes
generated successfully (16/16, 6 fallbacks, 0 failures, 40m40s, ~$1.80 spend).
But the eye-check was a catastrophic miss: **only 4/16 approved** (scenes 1, 2,
5, 6), 8 rejected (7-12, 15, 16), 2 marked regenerate (3, 4), 2 left pending
(13, 14). The closure criterion is ≥14/16 -- we missed by 10 scenes.

The user's rejection feedback (captured in `data/stories/story_2026_04_25_001/story.json`
via the S7 Phase 1 review UI) is consistent and unambiguous:

- *"the image looks like ai slop ... 2 people merging into one image and it doesn't capture what the prompt is trying to portray"* (scene 7)
- *"doesn't display what the prompt is saying. like there's no fog"* (scene 8)
- *"nothing like the prompt"* (scenes 9, 12)
- *"no dust or fog and the light is bright and everything is super visible"* (scene 10)
- *"looks like ai slop. doesn't look chained or drunk"* (scene 11)
- *"there is no forcing in the wall happening and the hand is all broken"* (scene 15)
- *"no cobwebs and looks ai generated"* (scene 16)

The pattern: **render quality is fine, content fidelity is broken.** Brightness
15-50 across all scenes (well above S6.3's recalibrated floor of 5.0); LAION
scores 4.6-6.8; subject edge density 0.005-0.087 on most scenes. The technical
pipeline produced clean output. But Flux Dev is generating "moody dark cinematic
atmosphere" that does not depict the **specific narrative beat** the prompt
describes. Characters merge, poses are missing (no chains, no drunkenness, no
walling-up action), atmospheric details from the prompt drop out (fog, cobwebs,
dust), and characters' faces drift between scenes (no continuity).

Diagnosis: the failure has four contributing causes, none addressed in S6.x:

1. **Prompt structure dilutes semantic conditioning.** The current
   `visual_prompts.j2` instructs Claude to produce "60-140 tokens, comma-
   separated descriptors. Lead with subject + setting, then lighting, then
   mood, then a quality clause." This produces keyword-soup prompts where the
   tail (`oil painting quality, fine film grain, deep crimson highlights`)
   competes with the narrative head (`man in jester motley`, `flambeau`,
   `niter-coated walls`) for Flux's limited attention budget. Flux resolves
   the conflict by leaning on its "moody catacomb" prior and dropping
   specifics.

2. **No structural language.** Prompts have zero camera framing (wide / medium /
   close), zero spatial layout (foreground / background), zero active-verb
   pose description. Flux invents these -- usually generic. That's why
   composition is wrong (people merging, no chains visible, no walling-up
   action).

3. **No reference conditioning.** PRD addition #3 (IP-Adapter FaceID +
   ControlNet Depth) was specified but never wired. Workflow ships without
   them. Result: Fortunato has a different face in every scene; composition
   has zero layout anchoring; specific actions get lost.

4. **1024x1024 square loses cinematic context.** Cask scenes depend on
   environment (catacomb depth, niche walls, descending staircase). Square
   framing crops these out. Compounding: the user's target distribution is
   Instagram reels (9:16 portrait), so every keyframe currently requires post-
   processing crop AND the wrong portion of the frame contains the visual
   content.

S7.1 fixes the four root causes simultaneously. We split the work into a
software-only Phase A (cheap, fast, no GPU required to validate code) and a
reference-conditioning Phase B (real engineering on the workflow + reference
management). The session ends with one A6000 verify run -- if eye-check ≥14/16,
S7 ships and S8 (Wan 2.2 video) is unblocked. If <14/16, S7.2 escalates to
`flux_pro_api` per the PRD's image_model escape hatch.

## 2. Decisions (six brainstorming forks)

| # | Question | Decision |
|---|---|---|
| 1 | Phase 1 (prompt rework) before Phase 3 (reference conditioning), or skip directly to Phase 3? | **Stack 1 + 3.** Prompt rework fixes a different problem (Flux interpreting "moody dark" instead of specific subject) than IP-Adapter / ControlNet (cross-scene face continuity + composition layout). Phase 1 alone is ~3 hours of code-only work; cost $0; risks nothing. Phase 3 alone leaves the prompt-structure root cause unaddressed -- IP-Adapter would feed a consistent face into a still-bad-content render. The user explicitly weighed quality > cost; both levers are required for the user's stated content-quality target. **Rejected:** B (skip to Phase 3 only) -- would burn $1.50 on a verify run that fails for the prompt-structure reason; C (Phase 1 only, defer Phase 3 to S7.2) -- would make a verify run that fixes content-fidelity gaps for single-figure scenes but still misses multi-figure / pose-specific scenes, leaving us in retry hell for ≥2 more sessions. |
| 2 | Aspect ratio | **768 x 1344 (≈9:16 portrait, Instagram reels target).** Closest Flux Dev-trained resolution to 9:16. Threads through to S8's Wan 2.2 video (which can produce 720x1280 portrait native; 768x1344 keyframes downscale cleanly). The track YAML's `video_model.base_resolution` flips from `[1280, 720]` to `[720, 1280]` in S8 -- registered now as a comment-only deferral, no code change in S7.1. Existing brightness / subject thresholds were calibrated on 1024x1024; with the aspect change we expect them to remain valid (per-pixel measurements; floor still 5.0 mean RGB; floor still 0.005 edge density). Will validate empirically on the verify run. **Rejected:** 720x1280 (exactly 9:16) -- not a Flux Dev-trained native resolution; produces visible artifacts. 832x1472 -- 1.226M pixels exceeds Flux's 1.0M-pixel sweet spot; slower per-candidate; minimal quality gain. |
| 3 | Reference conditioning approach | **Multi-ControlNet (OpenPose + Depth) + IP-Adapter FaceID, all three concurrent.** OpenPose handles character pose specificity (chained, masked, walling, drunk leaning); Depth handles scene-level spatial layout (foreground / background, niche depth, staircase descent); IP-Adapter handles cross-scene face continuity (Fortunato is the same man across all 16 scenes). The PRD #3 specified IP-Adapter + ControlNet Depth; we add ControlNet OpenPose because the user's specific failure feedback ("doesn't look chained or drunk", "hand is all broken") is pose-specific not depth-specific. **Compute cost:** multi-ControlNet roughly 2x per-candidate inference time on A6000 (~50s -> ~100s); accepted. Total Cask render projection: 16 scenes x 3 candidates x ~100s = ~80 min on A6000 (was ~40 min in S7 Phase 2). **Rejected:** Depth only (PRD baseline) -- doesn't address the pose feedback. OpenPose only -- doesn't address the spatial-layout feedback ("no fog", "everything super visible"). InstantID instead of IP-Adapter FaceID -- IP-Adapter is more mature with Flux Dev; InstantID still has open compatibility issues. T2I-Adapter -- lighter weight than ControlNet but worse content fidelity in current literature; we want quality over speed. |
| 4 | CLIP-similarity gate | **Yes, in-server. New `POST /clip-sim` endpoint** in score_server.py using the existing ViT-L-14 + CLIP-L model (already loaded for LAION-Aesthetics scoring). Cost per call: zero marginal (model already in VRAM); ~50ms / call. Inserted between the `subject_passed` gate and the LAION aesthetic scoring -- catches semantic-mismatch candidates BEFORE we waste the LAION call. **Per-track threshold** in YAML: `clip_min_similarity: 0.20` for atmospheric_horror; can be raised once we have empirical data. **Rejected:** Out-of-process Python helper (e.g., calling `open_clip` from keyframe_generator directly) -- would require keyframe_generator to load CLIP-L weights locally, which the architecture explicitly avoids ("HTTP-only" boundary between local orchestration and on-box GPU work). Skipping the gate entirely -- leaves the user eye-check as the only semantic filter; we observed 8/16 false-positives in S7 Phase 2. |
| 5 | Claude vision content gate | **Default ON for atmospheric_horror, default OFF for other tracks.** Per-track `quality_gates.content_gate` field accepts `"off"`, `"claude"`, `"claude_strict"`. `"claude"` rejects scenes with score <= 5/10; `"claude_strict"` <= 7/10. Cost: ~$0.005 per Claude vision call x 48 candidates per Cask render = $0.24 / run; cheap insurance against another 8/16 reject rate. **Rejected:** Default ON for all tracks -- premature; bright tracks (children's fables, sci-fi) haven't generated yet; calibration data missing. Default OFF for all -- defeats the user's "best accuracy" stated preference. CLIP-only -- CLIP similarity is faster but coarser than full vision; CLIP measures embedding overlap, Claude vision actually reads the image. Both are complementary: CLIP filters obvious mismatches cheaply; Claude catches subtle "the prompt asked for chains but I don't see chains" cases. |
| 6 | Reference image generation strategy (per-character face refs) | **Auto-generate via Flux + user approval through review UI.** New `CharacterReferenceStage` runs after visual_prompts: discovers characters from visual_prompts output, generates 3 candidate face refs per character with a deterministic prompt template, surfaces them in a new review UI page (`/story/<id>/characters`), user picks one (which sets `Story.characters[name] = path`). Picked references feed into IP-Adapter on every subsequent scene generation. **Rejected:** User provides face refs externally (Midjourney, stock, real photos) -- breaks the "everything pipelined from public-domain narrative input" principle; introduces personal-data / image-rights risk. Flux generates one ref deterministically (no review) -- removes user agency over the most identity-defining decision in the pipeline; if the auto-generated ref is wrong, all 16 scenes inherit the wrong face. Pick from a pool of pre-generated stock faces -- adds a curation problem we don't need; characters in literary horror benefit from per-story tonal consistency the auto-gen approach gives. |

### Explicit non-decision: `flux_pro_api` (deferred to S7.2)

PRD addition #5's BFL Pro API escape hatch remains the contingency if the
S7.1 verify run still misses the ≥14/16 target. Rationale unchanged from
S6.3:

- Bypasses cloud-first GPU validation that S8-S13 depend on.
- Different code path (model routing) -- proper home is a per-track
  `image_model.backend` flag with a dispatcher, not a one-off.
- We should run a **properly-conditioned** Flux Dev workflow before paying for
  hosted API. S7.1 is that properly-conditioned attempt -- new prompts,
  reference conditioning, multi-ControlNet. Only if S7.1 ALSO fails does
  `flux_pro_api` become the escape hatch.

### Explicit non-decision: video aspect ratio change in S8

Aspect ratio change to 768x1344 in S7.1 forces a downstream change in S8
(Wan 2.2 video gen base resolution flips from `[1280, 720]` to `[720, 1280]`).
We register the YAML field move in S7.1 but don't touch the video stage code.
The video stage's tests will fail explicitly when S8 starts -- that's a
visible signal we want, not a silent breakage we want to hide.

### 3.4 Reference image management contract

**Story-level state** (`Story.characters`):
```
{
  "Fortunato": "data/stories/story_2026_04_25_001/references/Fortunato/candidate_2.png",
  "Montresor": "data/stories/story_2026_04_25_001/references/Montresor/candidate_0.png"
}
```

**Per-scene state** (`Scene` fields):
```
character_refs: ["Fortunato", "Montresor"]  # both characters in this scene
pose_ref_path: "scene_007/_pose.png"
depth_ref_path: "scene_007/_depth.png"
composition_notes: "Medium shot. Two men face each other across a vault arch.
                    Foreground: Montresor in dark cloak holding a torch. Background:
                    Fortunato in jester motley, leaning against the wall."
```

**Resolution at render time** (`keyframe_generator.generate_for_scene`):
```python
face_ref = None
if scene.character_refs:
    primary = scene.character_refs[0]              # multi-char: first wins; S7.2 will improve
    face_ref = story.characters.get(primary)       # may be None if user hasn't picked
inject(workflow, face_ref_path=face_ref,
       depth_ref_path=scene.depth_ref_path,
       pose_ref_path=scene.pose_ref_path,
       ...)
```

When `face_ref is None` AND `scene.character_refs is non-empty` AND
`Story.characters.get(scene.character_refs[0]) is None`:
**`generate()` halts** with a clear error: `"scene N expects character 'X' but
Story.characters has no path for 'X'; run `platinum review characters` first"`.
This is a hard error, not a fallback -- the user's intent ("this scene contains
Fortunato") needs ref-picking before render.

## 7. Open questions and risks

**R1: Multi-ControlNet inference time on A6000.** Current single-Flux render
~50s per candidate at 1024^2 / 60 steps. Multi-ControlNet is roughly +50% per
extra ControlNet; with 2 ControlNets + IP-Adapter we project ~100s per
candidate. 16 scenes x 3 candidates x 100s = 80 min. If render time exceeds
~2 hours wall-clock, we should drop to one ControlNet + IP-Adapter for the
Phase B verify run; ship two-ControlNet as a follow-on if that closes the
content gap.

**R2: Reference image quality.** The auto-gen reference candidates (one per
character) will have the same Flux Dev limitations as scene candidates --
they may be on-theme but content-wrong. Phase B's review UI lets the user
reject all 3 and request regeneration; if all 3 are unusable, the user can
edit the per-character prompt and retry. Cost per regeneration: ~30 sec on
A6000 + ~$0.001 of compute time.

**R3: Aspect ratio threshold recalibration.** Brightness floor = 5.0 mean RGB
and subject floor = 0.005 edge density were both calibrated on 1024^2. They
are PER-PIXEL averages; aspect change should not affect them. We register
this as an "expect no change but verify" item; if the verify run halts on
brightness or subject for unexpected reasons, we recalibrate from the live
data (S6.2 / S6.3 pattern).

**R4: Claude vision gate API cost runaway.** Default-on for atmospheric_horror
costs ~$0.005 per call x 3 candidates x 16 scenes = $0.24 per Cask render.
If the user runs many iterations during eye-check tuning, costs add up. We
register this as a known cost; the `--no-content-gate` CLI flag offers a
quick-skip for debugging. If costs exceed ~$1 per session, consider
rate-limiting (e.g., only run content gate on candidates that pass CLIP +
LAION; not on candidates that already failed).

**R5: ControlNet preprocessor accuracy.** OpenPose and DepthAnythingV2 are
state-of-the-art preprocessors but they fail on edge cases (occluded
characters; non-human subjects; very dark frames). The Phase B
PoseDepthMapStage has no fallback for preprocessor failure -- if a scene's
prerender produces unusable pose / depth maps, the resulting reference
conditioning is garbage. We log preprocessor success rates; if <80% of
scenes get usable maps, S7.1.B iteration adds a "skip ControlNet for this
scene" override.

**R6: ComfyUI extension for IP-Adapter / ControlNet.** ComfyUI's extension
ecosystem is fragmented; IP-Adapter Flux support is in the
`ComfyUI-IPAdapter_plus` extension, ControlNet Flux support is in
`ComfyUI-Advanced-ControlNet`. `vast_setup.sh` will git-clone these into
`/workspace/ComfyUI/custom_nodes/`. If extension API changes break our
workflow JSON, we pin to specific commits in vast_setup.sh.

**R7: Workflow JSON validation drift.** Our workflow_signature pre-flight
check from S6.3 will catch box-side drift, but Phase B adds 8 new nodes to
the workflow -- we need to re-stamp the signature post-merge.
